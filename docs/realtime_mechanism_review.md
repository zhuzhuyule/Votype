## 二、必须补充的内容

### 2.1 验收标准（高优先级）

原文档未定义"功能正常"的判定条件。建议补充：

| 指标                | 阈值             | 说明                        |
| ------------------- | ---------------- | --------------------------- |
| 短音频 Partial 延迟 | < 500ms          | 1~3秒音频从触发到显示的时间 |
| 中音频 Partial 延迟 | < 1500ms         | 5~10秒音频的延迟上限        |
| 15秒边界行为        | 自动进入静默模式 | 验证超过15秒后不再触发推断  |
| CPU 占用率          | < 80%            | 持续推断时的峰值            |

### 2.2 测试计划（高优先级）

原文档未包含测试策略。建议补充：

**功能测试用例：**

- 短按场景（1~3秒）：验证 Partial 正确显示
- 长按边界场景（15秒）：验证自动进入静默模式
- 松手后终态交接：验证 Partial 与最终结果正确衔接
- 快速连续触发：验证互斥锁机制生效

**性能测试用例：**

- 长时间录音（30秒+）：验证无内存泄漏、无线程阻塞
- 多引擎对比：Whisper vs Parakeet vs SenseVoice 性能差异

### 2.3 回滚与降级策略（中优先级）

原文档未包含异常情况应对方案。建议补充：

**回滚条件：**

- 若 15 秒内延迟 > 2 秒的频率超过 30%，回滚至纯离线模式
- 若用户投诉率超过阈值，禁用模拟实时功能

**动态降级：**

- 检测到推断耗时 > 1 秒时，自动降低触发频率（从每1秒降至每2秒）
- 连续两次推断超时后，完全降级为离线模式，仅在松手时输出终态

## 三、建议补充的内容

### 3.1 前端状态机细化

原文档对前端状态交接的描述过于简略：

> "处理好录音终止时遗留 Partial 状态与最终完整结果的交接问题"

建议细化为具体的状态转换图：

```
[录音中] --每N秒触发--> [显示 Partial]
[录音中] --超过15秒--> [静默收集]
[录音结束] --最后一次推断--> [显示终态]
[显示 Partial] --收到终态--> [替换 Partial]
```

### 3.2 引擎差异化配置

不同引擎性能差异显著，建议在文档中明确：

| 引擎            | 预估延迟  | 建议触发间隔 |
| --------------- | --------- | ------------ |
| Whisper (base)  | 200~400ms | 1秒          |
| Whisper (small) | 100~200ms | 1秒          |
| Parakeet        | 150~300ms | 1秒          |
| SenseVoice      | 100~150ms | 0.8秒        |

### 3.3 竞态条件细节

原文档提到"跳 过这一轮推断"，但时序细节不足。建议补充：

- 音频缓冲区采用 Ring Buffer 或 Vec 追加模式
- 互斥锁粒度：锁住"推断任务"而非"音频收集"
- 跳过策略：记录"待处理但未处理"的音频累计量，松手时一并处理

### 6. 回滚与降级策略

#### 6.1 回滚条件

满足以下任一条件时，回滚至纯离线模式（仅在松手时输出终态）：

- 延迟 > 2 秒的频率超过 30%（按天统计）
- 用户反馈 "卡顿严重" 超过 5 次
- 内存泄漏确认（30 秒录音内存增长 > 200MB）

#### 6.2 动态降级规则

```
if (last_inference_duration > 1500ms) {
    trigger_interval = max(trigger_interval * 1.5, 2000ms);
}
if (last_inference_duration < 800ms && trigger_interval > 1000ms) {
    trigger_interval = 1000ms; // 恢复基准
}
```

#### 6.3 降级开关

在设置中添加 "模拟实时" 开关，默认开启。用户可手动关闭以获得稳定的离线体验。
