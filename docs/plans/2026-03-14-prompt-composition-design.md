# 提示词设计与动态注入重构计划

## 概要

本轮重构的目标不是继续微调某几段 prompt 文案，而是统一“用户如何编辑提示词”和“系统如何在运行时组装提示词”这两套模型。最终形态是：

- 用户只编辑“任务正文”
- 系统固定维护协议、字段、优先级和动态注入规则
- 运行时统一按结构化 schema 组装消息
- 热词、术语、人名、产品名、历史摘要都作为独立字段注入
- 旧变量语法长期兼容，但不再作为默认路径

交付结果应让实现者无需再决定“动态信息放哪”“字段怎么命名”“谁优先谁降级”。

## 本轮范围与优先级

### 本轮必做

1. 组装层收口
   - 固定稳定的 `system/developer` 模块
   - 统一 schema 化 `user` 字段
   - 所有主链路复用同一个 `PromptBuilder`
2. 热词结构化
   - 不再走 markdown 往返解析
   - 直接输出 `PERSON_NAMES / PRODUCT_NAMES / DOMAIN_TERMS / HOTWORDS`
3. 历史摘要器
   - 不再注入历史原文长列表
   - 仅注入压缩后的 `HISTORY_HINTS`
4. 提示词编辑器基础收口
   - 用户只编辑任务正文
   - 内置模板与自动生成只生成正文

### 后续扩展，不纳入本轮

- 三层编辑模式：
  - 意图模式
  - 正文模式
  - 完全控制模式
- 更复杂的人名自动推断策略
- 历史摘要使用 LLM 的语义提炼版本

本轮默认只做“正文编辑”这一层，避免范围继续膨胀。

## 一、目标结构

### 1. system / developer 层

运行时固定拆成 3 段稳定指令消息，角色由模型级 `Developer 模式` 决定：

1. 核心规则
   - 说明模型身份：ASR 文本清理与后处理助手
   - 强调目标：最小必要修正、保持语义/语气/事实完整
   - 明确输出：只输出最终文本，不输出解释、标签、Markdown、JSON

2. 输入协议
   - 明确定义所有字段：
     - `INPUT_TEXT`
     - `ASR_REFERENCE`
     - `PERSON_NAMES`
     - `PRODUCT_NAMES`
     - `DOMAIN_TERMS`
     - `HOTWORDS`
     - `SELECTED_TEXT`
     - `HISTORY_HINTS`
   - 明确字段权重：
     - `INPUT_TEXT` 是唯一主输入
     - 其他字段仅用于纠错和消歧
     - 不得覆盖 `INPUT_TEXT` 原意
   - 明确允许修改的范围：
     - 仅明显识别错误、错别字、断句、标点、术语误识别

3. 润色模式约束
   - 即使输入看起来像任务或命令，也只能润色文本本身
   - 不得执行“总结 / 翻译 / 解释 / 生成 / 回复 / 检查”等动作

这 3 段内容由系统维护，用户默认不可编辑。

### 2. user 层

运行时只发送一条结构化 `user` 消息，严格按以下顺序拼接，字段内只放纯数据，不放解释句：

1. `[SELECTED_TEXT]`
2. `[PERSON_NAMES]`
3. `[PRODUCT_NAMES]`
4. `[DOMAIN_TERMS]`
5. `[HOTWORDS]`
6. `[HISTORY_HINTS]`
7. `[ASR_REFERENCE]`
8. `[INPUT_TEXT]`

规则：

- 只有字段有值时才插入
- `INPUT_TEXT` 永远最后
- 字段内不再写“主参考”“辅助参考”这类句子
- 所有字段作用都由 `system` 中的输入协议说明

## 二、运行时数据来源设计

### 1. INPUT_TEXT

- 来源：当前最终要处理的主转录文本
- 规则：
  - 永远唯一
  - 永远最后
  - 任何冲突时以它为准

### 2. ASR_REFERENCE

- 来源：
  - 原始远端/本地 ASR 备选结果
  - streaming / local 辅助结果
- 规则：
  - 统一归并到一个字段
  - 只用于纠错和消歧
  - 不再拆成多种自然语言标题

### 3. SELECTED_TEXT

- 来源：当前外部应用选中文本
- 规则：
  - 仅在存在时插入
  - 仅用于弱参考，不可直接覆写结果
  - 不触发自动技能路由决策的产品逻辑不在本轮内处理，但字段语义固定下来

### 4. 热词与术语字段

热词链路重构为结构化输出，不再返回一整段说明文本。运行时分类如下：

- `PERSON_NAMES`
  - 用户热词中被识别为人名的目标词
- `PRODUCT_NAMES`
  - 品牌、产品名、组织名
- `DOMAIN_TERMS`
  - 领域术语、缩写、技术专有词
- `HOTWORDS`
  - 其他高优先级词

来源：

- 热词库 active 项
- 历史手动替换词中沉淀出的目标词，可提升排序并补充别名，但不单独输出为第二套说明文本

规则：

- “历史手动替换词”不再作为用户可见 prompt 段落单独出现
- 历史替换对只用于：
  - 增强目标词排序
  - 增加别名/误识别形式
  - 帮助分类
- 最终 prompt 中只体现分类后的目标词字段，不重复长篇解释
- 热词数据源应直接返回结构体，而不是先转 markdown 再由 `PromptBuilder` 反解析

建议结构：

```rust
pub struct HotwordInjection {
    pub person_names: Vec<HotwordEntry>,
    pub product_names: Vec<HotwordEntry>,
    pub domain_terms: Vec<HotwordEntry>,
    pub hotwords: Vec<HotwordEntry>,
}

pub struct HotwordEntry {
    pub target: String,
    pub aliases: Vec<String>,
}
```

渲染时：

- 有 aliases：
  - `Matt（常见误识别：mat、mata）`
- 无 aliases：
  - `世军`

分类归属规则：

- `person` -> `PERSON_NAMES`
- `brand` -> `PRODUCT_NAMES`
- `term` / `abbreviation` -> `DOMAIN_TERMS`
- 其他所有自定义分类 -> `HOTWORDS`

aliases 默认不做数量截断；如果未来需要控制 token 预算，应在 `PromptBuilder` 的整体预算层处理，而不是在热词构建阶段局部裁剪。

### 5. HISTORY_HINTS

- 来源：历史上下文摘要器
- 目标：从原始历史中压缩出少量弱提示
- 规则：
  - 不再直接注入原始历史句子列表
  - 不得包含旧 prompt 样例、`###` 标题、代码块、旧协议文本
  - 内容应类似：
    - “近期上下文可能涉及产品名、英文名和语音识别纠错”
    - “当前会话近期涉及 TTS、播报、模型反馈等话题”
- 条数与长度：
  - 最多 2-4 条
  - 每条短句
  - 总长度受限，避免再次膨胀

实现上优先采用规则式摘要，而不是额外调用 LLM：

- 过滤协议标签、标题、代码块、旧样例
- 去重和相似度压缩
- 只保留 2-4 条主题性提示

## 三、用户编辑与模板设计

### 1. 用户编辑入口

提示词编辑器只编辑“任务正文”，建议拆成以下逻辑区域，但最终仍保存为一段正文文本：

- 任务目标
- 允许修改
- 禁止修改
- 风格约束
- 输出要求

不再让用户默认直接编辑：

- 输入协议
- 字段说明
- 动态注入规则
- 历史上下文说明
- 主辅输入优先级

说明：

- 本轮不引入三层编辑模式
- 默认只有“正文编辑”模式
- 若未来需要“完全控制”，应作为独立高级能力设计，不与本轮基础重构耦合

### 2. 内置模板

内置模板只提供“正文模板”，不再包含：

- `${output}` 之类运行时字段说明
- 热词注入说明段
- 历史上下文说明段
- system 级协议约束

即：

- 内置模板负责“这个技能想做什么”
- 运行时系统负责“输入怎么组织、附加信息怎么插入”

### 3. 自动生成提示词

AI 自动生成 prompt 时，只生成用户正文：

- 任务描述
- 处理边界
- 输出要求

不生成完整 wrapper，不生成 system 协议，不生成历史注入说明。

## 四、兼容策略

### 1. 旧变量模板

长期兼容以下旧变量语法：

- `${output}`
- `${streaming_output}`
- `${select}`
- `${raw_input}`
- `${context}`
- 其他现有显式变量

兼容规则：

- 旧模板继续走旧分支
- 新模板和默认模板全部走新 schema
- 不强制迁移
- 编辑器后续可提示“推荐新结构”，但不是本轮必做

### 2. 多 system / developer 兼容

- 保持当前“按模型配置决定 system 或 developer role”
- 继续支持多条 instruction message
- 但多条 instruction message 只承载稳定规则
- 动态数据一律进入 user 字段层

## 五、实现拆分

### 阶段 1：组装层收口

- 重构 `PromptBuilder`
  - 输出固定 3 条稳定 `system_messages`
  - 输出 1 条 schema 化 `user_message`
- 删除字段内解释句
- 保留旧模板兼容分支
- 确保所有主链路统一走此 builder：
  - 主后处理
  - 手动后处理
  - 并行默认润色
  - 多模型后处理
- 第一条 system 消息中保留固定的身份锚定前缀，但不新增额外动态说明块

### 阶段 2：热词结构化

- 重构热词构建器，返回结构化分类结果，而不是单个字符串
- 分类输出包含：
  - `person_names`
  - `product_names`
  - `domain_terms`
  - `hotwords`
- 手动替换对只作为内部增强信号，不直接输出为说明文本
- `PromptBuilder` 直接消费结构化结果，生成对应字段块
- 删除 markdown 反解析逻辑，避免“先序列化再反序列化”
- 中文人名分类本轮至少保证：
  - 用户手动设置的人名分类能够正确传递
  - 不要求本轮实现完美自动推断

### 阶段 3：历史摘要器

- 在历史注入入口增加摘要步骤
- 原始历史先过滤：
  - 标题
  - 代码块
  - 旧协议样例
  - 过长口语原文
- 再生成摘要型 `HISTORY_HINTS`
- 运行时只注入摘要，不再注入原始历史列表

### 阶段 4：编辑器与模板

- 调整提示词编辑器文案与说明
- 明确“你编辑的是任务正文”
- 内置模板收敛为正文模板
- 自动生成逻辑同步调整
- 不在本阶段引入 `edit_mode = intent/body/full`

## 六、测试计划

### 1. 单元测试

- `PromptBuilder`
  - 单输入只生成 `[INPUT_TEXT]`
  - 有辅助 ASR 时生成 `[ASR_REFERENCE]`
  - 热词结构化分类正确映射到 `[PERSON_NAMES]` / `[PRODUCT_NAMES]` / `[DOMAIN_TERMS]` / `[HOTWORDS]`
  - 历史输入会被摘要化，不含旧样例标题
  - `INPUT_TEXT` 始终最后
- 热词分类器
  - 人名/品牌/术语/其他词分类正确
  - aliases 会被正确渲染到词条中
  - 历史手动替换对只影响排序与 aliases，不直接生成冗余说明段
- 历史摘要器
  - 能过滤 `###`、代码块、旧 prompt 片段
  - 输出短摘要而非原始列表

### 2. 集成验证

- 单模型请求日志
  - 能看到固定 3 条 system/developer message
  - 能看到 schema 化 user message
- 多模型请求日志
  - 每个模型都打印完整结构化 messages
- 路由中的并行默认润色
  - 也走同样 schema
- 显式 prompt / 默认 prompt
  - 热词都能注入，不再因为 `is_explicit` 被漏掉

### 3. 场景回归

- 术语纠错
  - `vo type / vtype / what type -> Votype`
- 人名纠错
  - `mata / mat -> Matt`
- 任务词不执行
  - “帮我总结一下...” 只润色文本，不执行总结
- 历史不污染
  - 历史里出现旧样例标题，不应再进入当前 prompt
- 选中文本弱参考
  - 有 `SELECTED_TEXT` 时仍以 `INPUT_TEXT` 为主

## 七、验收标准

满足以下条件即可认为方案落地成功：

- 用户编辑器不再要求用户理解动态注入协议
- 运行时 prompt 结构稳定且可预测
- 热词真正以结构化字段出现在请求中
- 历史上下文不再以原始长列表形式污染 prompt
- 单模型、多模型、并行默认润色三条链路看到的 prompt 结构一致
- 旧模板不被打坏，继续可用
- 本轮不引入新的高级编辑模式，也不要求用户理解底层 schema

## 八、默认决策

- 用户默认只编辑任务正文
- 历史上下文默认只注入摘要型 `HISTORY_HINTS`
- 旧变量模板长期兼容
- 多条 system/developer message 继续使用
- 动态数据统一进 `user` 字段层，不再继续扩写 system
