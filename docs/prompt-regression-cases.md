# 提示词最小回归样例

这份样例用于人工回归验证以下四类提示词链路：

1. 默认润色 `system_default_correction`
2. 技能路由 `system_skill_routing`
3. 热词注入 `build_llm_injection`
4. 纠错分析 `system_confidence_check` / `system_correction_analysis`

目标不是覆盖所有情况，而是用最少样本尽快发现以下回归：

- 过度润色
- 误路由
- 热词误命中
- 把普通润色误判成 ASR 错误

## 使用方法

每次修改相关提示词后，至少手工检查以下样例：

1. 取一条样例输入
2. 走对应链路
3. 对照“预期结果”判断是否偏离
4. 若偏离，优先修改 prompt，不要先调阈值

## A. 默认润色回归

### A1 口语噪音清理，不改语义

- 输入：
  `嗯这个接口今天先不改了啊，等明天再说。`
- 预期：
  `这个接口今天先不改了，等明天再说。`
- 关注点：
  允许删语气词，但不能改成更正式的重写版本。

### A2 中英混输保留

- 输入：
  `OpenAI 的 API latency 有点高，我们晚点再 retry 一次。`
- 预期：
  保留中英混合表达，只修正空格或标点，不翻译英文。
- 禁止：
  把 `latency` 改成 “延迟”，把 `retry` 改成 “重试”。

### A3 普通想法表达不能被过度优化

- 输入：
  `我在想这个方案是不是有点太复杂了。`
- 预期：
  原句保留或仅做轻微标点修正。
- 禁止：
  改成“这个方案可能存在过度设计问题”等重写表达。

### A4 热词纠正应命中

- 输入：
  `欧喷AI 那个接口最近不太稳定。`
- 前提热词：
  `OpenAI <- 欧喷AI`
- 预期：
  `OpenAI 那个接口最近不太稳定。`

### A5 热词证据不足时不要强替换

- 输入：
  `今天这个方案挺开放的。`
- 前提热词：
  `OpenAI <- 欧喷AI`
- 预期：
  不应把“开放”替换成 `OpenAI`。

## B. 技能路由回归

### B1 普通口述必须走 default

- 输入：
  `今天下午我得把那个接口再改一下。`
- 预期：
  `skill_id = "default"`
  `input_source = "output"`

### B2 带疑问语气仍然可能是 default

- 输入：
  `这个变量名是不是不太合适？`
- 预期：
  `skill_id = "default"`
- 说明：
  这是自然表达，不是明确请求 AI 执行动作。

### B3 有选中文本时命中 select

- 输入：
  `翻译这个`
- 前提：
  当前存在选中文本
- 预期：
  `input_source = "select"`
  路由到翻译类 skill，而不是 `default`

### B4 混合指令时命中 extract

- 输入：
  `帮我总结一下 今天讨论的重点是成本和排期。`
- 预期：
  非 `default`
  `input_source = "extract"`
  `extracted_content = "今天讨论的重点是成本和排期。"`

### B5 不能因为出现动作词就误路由

- 输入：
  `我刚才在想要不要总结一下这个问题。`
- 预期：
  `skill_id = "default"`
- 说明：
  用户在陈述想法，不是在发出命令。

## C. 热词命中回归

### C1 术语纠正

- 输入：
  `限量数据库的召回效果不太行。`
- 前提热词：
  `向量数据库 <- 限量数据库`
- 预期：
  `向量数据库的召回效果不太行。`

### C2 品牌纠正

- 输入：
  `扣波勒提斯那个集群现在还没起来。`
- 前提热词：
  `Kubernetes <- 扣波勒提斯`
- 预期：
  命中 `Kubernetes`

### C3 语义润色不能当热词

- 输入：
  `这个方案不太好。`
- 预期：
  不应凭空替换为任何热词或术语。

### C4 人名替换要受语境约束

- 输入：
  `这个名字听起来像小明。`
- 前提热词：
  某个企业术语或品牌名与“小明”发音接近
- 预期：
  不应仅凭音近强行替换。

## D. 纠错分析回归

### D1 明显 ASR 错误

- 原文：
  `欧喷AI 的接口`
- 润色后：
  `OpenAI 的接口`
- 预期：
  `is_hotword = true` 或 `type = "asr_error"`
  `category = "brand"`

### D2 术语修正

- 原文：
  `限量数据库`
- 润色后：
  `向量数据库`
- 预期：
  `is_hotword = true` 或 `type = "asr_error"`
  `category = "term"`

### D3 纯格式修正

- 原文：
  `OpenAI的API很好用`
- 润色后：
  `OpenAI 的 API 很好用`
- 预期：
  `is_hotword = false` 或 `type = "semantic_edit"`
- 说明：
  这是空格与格式规范化，不应入热词。

### D4 语气词清理

- 原文：
  `嗯这个方案啊我觉得还行`
- 润色后：
  `这个方案我觉得还行`
- 预期：
  `is_hotword = false` 或 `type = "semantic_edit"`

### D5 表述优化

- 原文：
  `这个方案不太好`
- 润色后：
  `这个方案不够理想`
- 预期：
  `is_hotword = false` 或 `type = "semantic_edit"`

## 建议观察指标

每轮 prompt 调整后，建议至少记录这四项：

- `default` 路由是否更保守
- 热词命中是否更准，误命中是否下降
- 默认润色是否仍然过度书面化
- 纠错分析是否把格式优化误判为热词

## 通过标准

最小通过标准：

- A1-A5 全部符合预期
- B1-B5 至少 4 条符合预期，且不能误伤 B1/B2/B5
- C1-C4 至少 3 条符合预期，且不能误伤 C3
- D1-D5 至少 4 条符合预期，且 D3/D4 不能误判为热词
