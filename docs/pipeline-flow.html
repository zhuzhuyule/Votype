Phase 1 — 录音

松开快捷键 → ASR 处理 overlay: transcribing

同时保存音频到数据库（pre-save）
0.5~3s
Phase 3 — 后处理 Pipeline
进入 unified_post_process() 立即切换 overlay: llm
修复前：overlay 仍显示"转录中"，直到实际 LLM 调用才切换
修复后：进入 pipeline 第一行就显示"润色中"
pipeline.rs:57 → show_llm_processing_overlay()
Step 1: 历史缓存匹配
短文本 + 启用 Smart Routing → 查 SQLite 是否有完全匹配的润色结果
<1ms
Step 2: Intent 意图分析 LLM 调用
短文本时调用 Intent 模型，分类为 PassThrough / LitePolish / FullPolish
同时判断 needs_hotword、language、token_count
1~2s
⑂ Intent 分支
PassThrough (无需润色)
直接返回原文
不调用 LLM
→ 跳到 Phase 4
LitePolish / FullPolish
继续 Step 3-4
选择模型 → 执行润色
Step 3: 模型选择
LitePolish → 轻量模型 + lite prompt
FullPolish → 完整模型 / Multi-Model 并发
<1ms
Step 4: 执行 LLM 润色 LLM 调用
构建 Prompt（注入 hotword、session context、scenario hint）
调用 LLM API，timeout 10s，支持 fallback 链
1~8s
Phase 4 — 结果处理
Pipeline 返回结果 <1ms
PipelineResult 枚举：Skipped / Cached / PassThrough / SingleModel / MultiModel
⑂ should_review 分支 — 这决定了后续行为
should_review = true (默认 Auto)
1 写数据库 <5ms
2 打开 Review Window
3 隐藏 Overlay
4 等待用户确认 ← 这就是"间隔"
5 用户按 Enter / 点击 → 粘贴
should_review = false (Never)
1 写数据库 <5ms
2 run_on_main_thread
3 隐藏 Overlay + 立即粘贴

无额外等待

关于"润色中"之后的等待：
如果你的 App Policy 是 Auto（默认），润色完成后会打开 Review Window 等待你确认。 这个"间隔"就是 Review Window 在等你按 Enter 或点击确认按钮。
如果你的 App Policy 是 Never，润色完成后会立即粘贴，中间不应有明显延迟。
