# 实时转录机制分析与方案设计

**日期：** 2026-02-28
**主题：** 在移除 Sherpa 引擎后，基于现有非流式引擎恢复实时文字上屏体验的机制分析与设计。

## 1. 机制背景与差异对比

之前 `main` 分支在集成 Sherpa 引擎时，利用了 Sherpa 底层 C++ API 提供的**原生流式识别 (Native Streaming)** 能力。

- **原生流式特征：** 引擎可以一帧一帧地接收音频数据。由于引擎内部维持着声学推断的状态，每次喂入新音频片段时，它只需接着之前的状态继续后向解码。
- **性能评估：** 这种方式的计算复杂度为 $O(N)$，且每次增量计算开销极低。无论是极短的音频还是长达数分钟的音频，均能顺滑地实时输出 Partial（部分）结果，系统资源消耗平稳。

而目前代码库在去除了 Sherpa 之后，仅保留了 Whisper、Parakeet、SenseVoice、Paraformer 等引擎。这些引擎主要被设计为**非流式（Offline）**的端到端模型。

- **非流式特征：** 模型不保存历史解码状态。它必须接收一整段完整的音频才能给出推断结果。当前 `main` 分支的默认机制也是在用户结束录音时，才将所有音频送入模型进行一次“全量一波算”。

## 2. 模拟流式效果：分段重复识别 (Segmented Offline Recognition)

为了在长按快捷键说话的过程中让用户的屏幕上能够“实时”跳出文字（典型的语音输入法体验），我们提出了基于这些非流式引擎的机制增强方案：**“分段重复识别”**。

### 2.1 机制原理

在后台开启一个独立线程累计接收录音音频。随着音频时间推移（如第 1 秒、第 2 秒、第 3 秒等），每当积攒了固定步长的新音频，就强行**将从头开始到当前时刻的所有音频作为一个完整的切片**，送入离线引擎重新进行一次从零开始的完整转录推断。
获取结果后，发送事件给前端，前端将其作为“部分结果 (Partial)”展示。

### 2.2 合理性与隐患评估

- **合理之处 (适用性)：**
  针对短片段目标场景（用户按住快捷键说话，大部分情况下在一两句话、10~15秒内即可说完），现代硬件（特别是 Apple Silicon M 系列芯片）的算力十分充沛。使用 Whisper 或 Parakeet 推断几秒钟的离线音频一般只需 100~300 毫秒。这种低延迟足以欺骗人的感官，让用户以为文字是随着说话实时蹦出来的，极大提升了用户体验。这也是在当前技术栈内不引入新架构的唯一有效解决方案。

- **不合理之处 (性能隐患)：**
  因为模型不保存状态，随着录音越来越长，每次重新推断的包袱越来越重。
  计算复杂度随时间推移呈 $O(N^2)$ 级别的退化。例如，录音到第 15 秒时，引擎需要重新处理前 14 秒已经被计算过数十遍的旧音频。如果音频长达 1 分钟，不仅发热量剧增，单次推断时间也可能超过 1 秒钟，从而导致前端 Partial 事件获取滞后、排队堆积以及系统严重的卡顿乃至死锁。

## 3. 核心防护策略与状态流转

鉴于“分段重复识别”在短时间场景下的高收益和在长时间场景下的高风险，实施此方案必须包裹严密的保护与降级策略。

### 3.1 前端状态机流转

前端的 UI 状态流转需严格匹配后端的保护机制：

```text
[录音中] --每N秒触发--> [显示 Partial]
[录音中] --超过15秒--> [静默收集 (不再发 Partial)]
[录音结束] --最后一次推断--> [计算终态]
[显示 Partial] --收到终态--> [替换 Partial]
```

### 3.2 并发互斥与跳帧机制

后台线程不应机械地根据时间无脑触发推断。

- 必须使用互斥锁（如 `try_lock`）锁住**推断任务**本身。
- 如果上一任务耗时太长还未释放锁，这一轮新进来的音频帧应通过 Vec 追加模式直接拼接到缓冲区，直接**跳过**这一轮的推断触发，累计到下一次处理。这样彻底避免了由于推断速度慢于输入速度而导致的任务队列堆积。

## 4. 验收标准与测试计划

### 4.1 功能与性能验收

| 测试场景         | 预期行为 / 指标  | 说明                                           |
| ---------------- | ---------------- | ---------------------------------------------- |
| 短音频 Partial   | 延迟 < 500ms     | 1~3秒音频从触发到显示的时间                    |
| 中音频 Partial   | 延迟 < 1500ms    | 5~10秒音频的延迟上限                           |
| 15秒边界测试     | 自动进入静默模式 | 验证超过 15 秒后不再触发推断，仅在松手时出结果 |
| CPU 异常升高保护 | CPU 占用率 < 80% | 持续高频推断时的峰值控制                       |
| 连续快速短句触发 | 无错乱覆盖现象   | 验证互斥锁机制与跳帧逻辑正确生效               |

### 4.2 引擎差异化基准测试

不同模型的建议基准：

- **Whisper (base/small):** 预期延迟 100~400ms，建议触发间隔：1秒
- **Parakeet:** 预期延迟 150~300ms，建议触发间隔：1秒
- **SenseVoice:** 预期延迟 100~150ms，建议触发间隔：0.8秒

## 5. 动态降级与回滚策略

为了防止极端设备或过长录音导致的系统崩溃，必须实现运行时的弹性降级机制：

**动态触发降级策略：**

1. **轻度降级：** 检测到上一次推断耗时 > 1500ms 时，自动降低触发频率（如从每 1 秒降至 `max(原间隔 * 1.5, 2000ms)`）。
2. **重度降级：** 若连续 2~3 次推断超时（如耗时 > 2000ms），则**彻底降级为离线模式**，放弃 Partial 更新，仅在松手时做最后一次全量推断。

**回滚条件（可配置开关）：**
后续可在设置中提供显式的“模拟实时体验”开关，允许用户在设备性能不足或频繁触发降级的情况下，主动关闭此功能回退到纯离线模式。
